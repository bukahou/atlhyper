# AtlHyper 下一阶段规划

> 核心功能（L1~L4）已基本完成，本文档规划「让平台更好用、更可靠、更完整」的外壳能力。

| 能力 | 状态 |
|------|------|
| K8s 全资源观测（20+ 资源类型） | ✅ |
| 六大信号源（Metrics/APM/Logs/SLO/Events/K8s） | ✅ |
| 因果拓扑图 + 依赖传播 | ✅ |
| AIOps 引擎（基线/异常/风险/事件/关联） | ✅ |
| AI 增强（多 LLM、Tool Calling、安全约束） | ✅ |
| GitOps CD 管道（GitHub App + Kustomize + 自动部署） | ✅ |
| 权限体系（JWT + RBAC 三级 + 审计日志） | ✅ |
| 通知分发（Slack + Email） | ✅ |
| 指令执行框架（Pod/Deployment/Node 操作） | ✅ |
| 预算控制（日/月限额 + 触发模式 + Fallback） | ✅ |

### 缺失的外壳能力（按优先级分层）

---

## Phase 1：体验完善（让现有功能更好用）

> 不增加新功能，打磨已有功能的易用性和可靠性。

### 1.1 Dashboard 可定制化

**现状**：Overview 页面是固定布局，用户无法选择关注哪些指标。

**目标**：
- 支持卡片拖拽排序 / 显示隐藏
- 用户偏好持久化到 localStorage 或后端

**工作量**：中

### 1.2 全局搜索

**现状**：只有仓库列表有搜索，其他页面没有统一搜索入口。

**目标**：
- 顶部全局搜索栏（Cmd+K 快捷键）
- 搜索范围：Pod 名、Service 名、Incident ID、Node 名
- 搜索结果直接跳转到对应详情页

**工作量**：中

### 1.3 时间范围选择器统一

**现状**：各页面的时间范围选择器实现不统一。

**目标**：
- 统一的 TimeRangePicker 组件
- 支持预设（最近 1h/6h/24h/7d）+ 自定义范围
- 全局共享时间范围（切换一处，全局生效）

**工作量**：小

### 1.4 操作确认与反馈增强

**现状**：部分危险操作（Pod 重启、Node Cordon）确认流程较简单。

**目标**：
- 危险操作二次确认弹窗（显示影响范围）
- 操作执行后实时状态跟踪（而非只显示"已发送"）
- 操作历史可追溯（已有 Command 历史，需前端展示优化）

**工作量**：小

---

## Phase 2：可观测性补全（补齐告警闭环）

> 让 AIOps 引擎的能力真正落地到用户可感知的告警。

### 2.1 告警规则引擎

**现状**：AIOps 能检测异常，但用户无法自定义告警规则。

**目标**：
- 基于指标阈值的告警规则（CPU > 90% 持续 5 分钟）
- 基于日志关键词的告警规则（error/panic/OOMKilled）
- 基于 SLO 违规的告警规则（可用性 < 99.9%）
- 告警规则 CRUD + 静默/抑制

**工作量**：大
**依赖**：通知模块已就绪

### 2.2 日志异常自动检测

**现状**：日志可查询但无异常自动检测。

**目标**：
- Error/Fatal 日志自动聚类
- 日志量突增检测（对接基线模块）
- 与 Incident 自动关联

**工作量**：中

### 2.3 Trace 延迟异常告警

**现状**：APM 拓扑完整，缺延迟突增告警。

**目标**：
- P95/P99 延迟基线 + 异常检测
- 慢请求自动关联到上下游服务
- 与 Incident 自动关联

**工作量**：中

---

## Phase 3：多集群支持（从单集群走向多集群）

> 当前最大的架构限制。解除后可管理多个 K8s 集群。

### 3.1 多 Agent 注册与管理

**现状**：单集群单 Agent，Agent 地址硬编码。

**目标**：
- Agent 自动注册（Agent 主动上报到 Master）
- Agent 健康管理（心跳超时 → 告警）
- Agent 版本管理（显示版本、提示升级）
- 前端集群切换器（顶部导航栏）

**工作量**：大
**风险**：需要改动 AgentSDK 通信协议

### 3.2 多集群数据隔离

**现状**：DataHub 按 ClusterID 隔离，但 UI 和查询层未做多集群切换。

**目标**：
- 所有 API 查询支持 clusterId 参数
- 前端集群选择器绑定到所有页面
- 跨集群概览页（所有集群健康度一览）

**工作量**：大

---

## Phase 4：L5 自主控制（闭环自愈）

> 从「观测 + 告警」升级到「观测 → 推理 → 决策 → 执行」。

### 4.1 自愈策略引擎

**现状**：Command 框架完整（能执行 Pod 重启、扩缩容等），但缺自动触发逻辑。

**目标**：
- 策略定义：`当 X 条件满足时，执行 Y 操作`
- 策略示例：
  - Pod CrashLoopBackOff 超过 3 次 → 自动重启
  - CPU > 95% 持续 10 分钟 → 自动 HPA 扩容
  - Node 磁盘 > 90% → 告警 + 自动清理临时文件
- 策略审批：高危操作需人工审批（非全自动）
- 策略执行日志：谁触发、什么条件、执行结果

**工作量**：大
**前置**：告警规则引擎（Phase 2.1）

### 4.2 变更影响预评估

**现状**：操作直接执行，无事前影响分析。

**目标**：
- 执行操作前，AI 分析影响范围（依赖图 + 历史数据）
- 显示风险等级和建议

**工作量**：中

---

## Phase 5：生产加固（运维可靠性）

> 让平台本身也是可靠的。

### 5.1 Master 高可用

**现状**：单实例 Master，挂了就全挂。

**目标**：
- Master 无状态化（SQLite → PostgreSQL/MySQL 可选）
- 支持多副本部署
- DataHub 内存缓存同步（或切换为 Redis）

**工作量**：大

### 5.2 数据保留策略

**现状**：ClickHouse 数据无自动清理，SQLite 审计日志无限增长。

**目标**：
- ClickHouse TTL 配置（Metrics/Logs/Traces 保留天数可配）
- SQLite 审计日志定期归档
- 事件历史保留策略

**工作量**：小

### 5.3 备份与恢复

**现状**：无备份机制。

**目标**：
- SQLite 数据库定时备份（配置、用户、事件）
- 一键恢复

**工作量**：小

---

## Phase 6：生态拓展（可选方向）

> 根据用户反馈和实际需求选择性实现。

| 方向 | 说明 | 优先级 |
|------|------|--------|
| **Webhook 通知** | 除 Slack/Email 外支持自定义 Webhook | 低 |
| **API Token 认证** | 除 JWT 外支持长期 API Token（给 CI/CD 用） | 低 |
| **Helm Chart 部署** | 打包为 Helm Chart，一键部署 Master + Agent | 中 |
| **插件机制** | 自定义数据源 / 告警渠道 / AI Provider | 低 |
| **移动端适配** | 响应式布局 + PWA 推送 | 低 |
| **多语言扩展** | 英文支持 | 中 |

---

## 优先级总览

```
Phase 1  体验完善        ← 现在可以做，改动小，效果明显
Phase 2  告警闭环        ← 核心价值提升最大，让 AIOps 真正可用
Phase 3  多集群          ← 架构升级，解锁规模化
Phase 4  自主控制        ← 终极目标（L5），依赖 Phase 2
Phase 5  生产加固        ← 生产部署必须，可与 Phase 1-2 并行
Phase 6  生态拓展        ← 按需选做
```
